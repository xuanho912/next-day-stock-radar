# User Logic Review

来源：用户提供的 DOCX 逻辑文档。

结论：文档里最有价值的核心是“股票上涨来自正向预期差”，这一点已经被吸收到本项目的 `expectation_gap_score`、`confluence_score`、`payoff_quality_score` 和代理审查层里。

## 可直接吸收的部分

- 预期差是核心，不是“好公司”本身。
- 市场环境必须先过滤，个股不能脱离大盘风险偏好。
- 个股需要判断资金是否确认，不能只看基本面。
- 上涨空间必须和下跌风险一起看，不能只看胜率。
- 每只候选必须有失效条件，否则容易变成主观扛单。
- 规则模型先于机器学习，便于审查、解释和迭代。

## 需要改造的部分

文档原逻辑更适合 20/60 日 Alpha 评分模型，而本项目是次日高弹性雷达。两者目标不同：

- 20/60 日模型重视基本面成长、估值、盈利质量和中期超额收益。
- 次日雷达重视催化、成交量、价格结构、资金攻击路径、风险闸门和可执行触发价。

因此，本项目不把长期基本面评分放在第一优先级。基本面只作为事件催化、预期差和风险扣分的输入。

## 必须排除的部分

文档中出现了“买入”“仓位”“止损”“清仓”等交易执行措辞。项目安全边界要求：

- 不输出自动交易。
- 不输出买入/卖出指令。
- 不输出仓位建议。
- 不承诺必涨必跌。

本项目只输出概率路径点位：

- `upside_trigger_level`
- `invalidation_level`
- `downside_risk_level`
- `next_day_expected_range`

这些是观察和验证用的路径标记，不是交易指令。

## 已落地到系统的改造

- `expectation_gap_score`：衡量催化、价格、成交是否形成真正超预期。
- `payoff_quality_score`：衡量预期上行和下行风险是否值得继续观察。
- `execution_quality_score`：衡量触发价、流动性和风险是否可执行。
- `precision_gate`：候选必须通过多维共振，不能只靠热度。
- `radar_agent_review.py`：用 agency-style 代理审查市场路径、主线、预期差、执行质量、风险和验证纪律。
- Forecast Ledger：每天记录预测，之后回填真实表现，避免用事后解释替代验证。

## 后续可作为 Challenger 的方向

可以单独做一个中期 Alpha Challenger，但不能替代次日雷达 Baseline：

- `stock_alpha_20d_challenger_v1`
- 目标：未来 20 日是否跑赢 SPY/QQQ。
- 输入：基本面成长、盈利质量、估值、技术趋势、资金、催化、市场/行业环境。
- 输出：只作为 shadow 研究，不影响次日 Top Candidates。

只有它在 30-60 个未来交易日里证明对次日候选质量也有增益，才允许把部分因子并入 Baseline。
