# Agency Agents Workflow

本项目参考 `msitarzewski/agency-agents` 的工作方法，但不直接引入它作为运行时依赖。

原因很简单：`agency-agents` 的核心价值不是某个股票预测算法，而是把复杂任务拆成专门角色，每个角色都有明确使命、证据要求、失败条件和质量闸门。对本项目来说，这比堆更多指标更有用。

## 如何落到本项目

每日流水线仍然是确定性数据工程：

1. 收集行情、新闻、事件、市场背景。
2. 生成候选股票和概率路径。
3. 排名并写入 Forecast Ledger。
4. 回填验证指标。
5. 生成 Dashboard JSON。
6. 运行 `radar_agent_review.py` 做代理式质量审查。
7. 前端只读取静态 JSON，不接触任何 API key。

## 代理角色

代理配置在 `config/radar_agents.json`。

- 市场路径代理：判断当前市场是否允许筛选高弹性机会。
- 板块主线代理：确认是否有资金主线和候选类型共振。
- 预期差代理：检查催化、价格、成交是否真正共振。
- 执行质量代理：检查触发价、失效价、赔率和流动性。
- 风险现实校验代理：专查旧数据、proxy、小样本、冲高回落和流动性风险。
- 验证代理：维护 Forecast Ledger 和 Baseline/Challenger 纪律。
- 数据质量代理：检查 provider 状态、降级和 stale warning。
- 中文仪表盘代理：确保第一屏先给结论，不暴露 key，不做交易指令。

## 输出文件

每日运行后会额外生成：

- `frontend/public/radar-agent-review.json`
- `outputs/radar_agent_review.json`
- `outputs/radar_agent_review.md`

Dashboard 主 JSON 也会包含 `agency_review` 字段。

## 质量闸门

`agency_quality_gate` 只有三个结果：

- `通过`：市场、数据、预期差、共振、赔率和执行质量同时达标。
- `谨慎通过`：存在数据降级、验证样本不足、proxy 或小样本警告，但候选仍可作为观察雷达。
- `不通过`：数据缺失、市场防守、无候选、流动性风险或关键代理失败。

## 产品边界

这层审查不会输出买入、卖出或仓位建议。它只回答：

- 今天是否适合筛选次日高弹性机会；
- 哪些候选真正共振；
- 哪些风险不能忽略；
- 当前模型是否已经被前向验证支持；
- 页面是否把不确定性讲清楚。

最终目标仍然是：宁可少给股票，也不要乱给；宁可输出 3 只真正共振的机会，也不要输出 20 只看起来专业但没有交易价值的股票。
